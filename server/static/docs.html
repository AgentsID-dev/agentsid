Quick Start

Follow these five steps to go from zero to fully protected MCP server.

Step 1: Install the SDK

See the Installation section above for your language.

Step 2: Create a project

npx agentsid init "My App"

This creates a project and prints your API key. Store it securely -- you will not see it again.

Step 3: Register an agent

import { AgentsID } from '@agentsid/sdk';

const aid = new AgentsID({
  projectKey: 'aid_proj_...'
});

const { agent, token } = await aid.registerAgent({
  name: 'research-assistant',
  onBehalfOf: 'user_abc',
  permissions: ['search_memories', 'save_memory'],
  ttlHours: 24
});

console.log('Agent ID:', agent.id);
console.log('Token:', token);

Agent Identity

Every agent gets a unique, verifiable identity backed by an HMAC-signed token. This is not a shared API key -- it is a specific identity for a specific agent instance, traceable to the human who authorized it.

Human user_abc
→
Agent research-assistant
→
Token aid_tok_...
→
Validate HMAC-SHA256
Agent tokens are self-validating -- signature verification requires no database call.

Permissions

AgentsID uses a deny-first permission engine. If no rule explicitly allows an action, it is denied. Deny rules always take priority over allow rules.

Step 1 Check Deny Rules
→
Step 2 Check Allow Rules
→
Step 3 Default Deny
Deny rules are always evaluated first. If nothing explicitly allows, the action is denied.

Pattern Matching

Tool patterns support Unix-style wildcard matching. Use * to match any characters.

PatternMatchesDoes Not Match
save_memorysave_memorysave_note, delete_memory
save_*save_memory, save_note, save_filedelete_memory
*_memorysave_memory, delete_memorysave_note
*Everything(nothing excluded)
 
await aid.setPermissions(agent.id, [
  { toolPattern: 'search_*',  action: 'allow' },
  { toolPattern: 'save_memory', action: 'allow',
    conditions: { category: ['note', 'preference'] } },
  { toolPattern: 'delete_*',  action: 'deny', priority: 10 },
]);

// Check a permission
const result = await aid.checkPermission(agent.id, 'delete_memory');
// result.allowed = false
// result.reason = "Denied by rule: delete_*"
await aid.set_permissions(agent.id, [
    {"tool_pattern": "search_*",  "action": "allow"},
    {"tool_pattern": "save_memory", "action": "allow",
     "conditions": {"category": ["note", "preference"]}},
    {"tool_pattern": "delete_*",  "action": "deny", "priority": 10},
])

# Check a permission
result = await aid.check_permission(agent.id, "delete_memory")
# result.allowed = False
# result.reason = "Denied by rule: delete_*"
aid.set_permissions(agent.id, [
  { tool_pattern: 'search_*',  action: 'allow' },
  { tool_pattern: 'save_memory', action: 'allow',
    conditions: { category: ['note', 'preference'] } },
  { tool_pattern: 'delete_*',  action: 'deny', priority: 10 },
])

# result.allowed = false

    new Rule("search_*",  "allow"),

        Map.of("category", List.of("note", "preference"))),
    new Rule("delete_*",  "deny", 10)
));

// Check a permission
CheckResult result = aid.checkPermission(agentId, "delete_memory");
// result.isAllowed() = false

Advanced Permission Controls

Permission rules support optional advanced fields for fine-grained control beyond simple allow/deny. These fields are set via the Set Permissions endpoint.

Resource Scoping

Use the conditions field to constrain which parameters a tool call can use. All conditions must match (AND logic). Values can be a single value or a list of allowed values.

{
  "tool_pattern": "save_memory",
  "action": "allow",
  "conditions": { "category": ["note", "preference"] }
}

Time-based Permissions

Use the schedule field to restrict a rule to specific hours and days. If the current time is outside the window, the rule is skipped during evaluation.

FieldTypeDescription
hours_startintegerStart hour, inclusive (0-23)
hours_endintegerEnd hour, exclusive (0-24)
timezonestringIANA timezone name (default: UTC)
daysstring[]Days of week: mon, tue, wed, thu, fri, sat, sun

Rate Limiting

Apply a sliding-window rate limit to an allow rule using the rate_limit field. When exceeded, the check returns allowed: false with a rate limit reason.

FieldTypeDescription
maxintegerMaximum calls in window (1-100,000)
perstringWindow size: second, minute, hour, or day

Data Classification

Use the data_level field to restrict which classification levels a tool can access. Valid levels: public, internal, confidential. If the tool call does not include a data_level parameter, the check passes regardless.

Approval Gates

Set requires_approval: true to hold matching actions for human review. The permission check returns pending_approval: true instead of allowed: true. Use the Approvals endpoints to approve or reject.

Combined Example

A single rule can combine all advanced fields:

{
  "tool_pattern": "transfer_funds",
  "action": "allow",
  "conditions": { "currency": ["USD", "EUR"] },
  "schedule": {
    "hours_start": 9,
    "hours_end": 17,
    "timezone": "US/Eastern",
    "days": ["mon", "tue", "wed", "thu", "fri"]
  },
  "rate_limit": { "max": 5, "per": "hour" },
  "data_level": ["confidential"],
  "requires_approval": true,
  "priority": 100
}
Evaluation order.

Advanced fields are evaluated after pattern matching: schedule is checked first, then rate limit, then data level, then approval gate. If any check fails, the rule does not apply.

Delegation Chains

Every agent action is traceable to a human. When agents delegate to sub-agents, permissions can only narrow at each hop -- they can never expand.

Human user_abc
read, write, delete
→
Agent A coordinator
read, write
→
Agent B sub-researcher
read
Permissions narrow at each delegation hop. Agent B cannot receive "write" because Agent A chose not to delegate it.
// Parent agent delegates to a child with narrowed permissions
const child = await aid.delegate({
  parentAgentId: 'agt_7x9k2mNpQ4rS1tUv',
  parentToken: 'aid_tok_...',
  childName: 'sub-researcher',
  childPermissions: ['search_memories'], // subset of parent's
  ttlHours: 12
});

// child.agent.id = "agt_newChildId"
// child.token = "aid_tok_childToken..."
# Parent agent delegates to a child with narrowed permissions
child = await aid.delegate(
    parent_agent_id="agt_7x9k2mNpQ4rS1tUv",
    parent_token="aid_tok_...",
    child_name="sub-researcher",
    child_permissions=["search_memories"],  # subset of parent's
    ttl_hours=12,
)

# child.agent.id = "agt_newChildId"
# child.token = "aid_tok_childToken..."
# Parent agent delegates to a child with narrowed permissions
child = aid.delegate(
  parent_agent_id: 'agt_7x9k2mNpQ4rS1tUv',
  parent_token: 'aid_tok_...',
  child_name: 'sub-researcher',
  child_permissions: ['search_memories'],
  ttl_hours: 12
)
DelegateResult child = aid.delegate(
    "agt_7x9k2mNpQ4rS1tUv",
    "aid_tok_...",
    "sub-researcher",
    List.of("search_memories"),
    12
);
Scope narrowing is enforced server-side.

If a child requests a permission the parent does not have, the delegation is rejected with HTTP 403.

Audit Trail

Every token validation and permission check is recorded in an append-only audit log. The log is cryptographically chained using SHA-256, making tampering detectable.

ALLOW save_memory agt_7x9k...Q4rS 14:35:12
DENY delete_memory agt_7x9k...Q4rS 14:35:28
ALLOW list_categories agt_abc1...f456 14:36:01
DENY save_memory agt_abc1...f456 14:36:15 
// Query audit log for a specific agent

Hash Chain Integrity

Entry 1 hash(genesis + data)
→
Entry 2 hash(prev + data)
→
Entry 3 hash(prev + data)
→
Entry N hash(prev + data)
Each entry's hash covers the previous entry's hash, forming a tamper-evident chain. Modifying any entry breaks all subsequent hashes.

API Reference

Base URL: https://api.agentsid.dev/api/v1 (hosted) or http://localhost:8000/api/v1 (self-hosted).

All endpoints (except project creation and health) require a project API key in the Authorization header:

Authorization: Bearer aid_proj_<your_project_key>

Projects

POST /api/v1/projects/ Create a new project and receive an API key ▸

Rate limit: 5 requests per minute per IP. No authentication required.

Request Body

FieldTypeRequiredDescription
namestringYesProject name (1-255 characters)
emailstringNoOwner email address

Response 201 Created

{
  "project": {
    "id": "proj_a1b2c3d4e5f6",
    "name": "my-app",
    "plan": "free",
    "created_at": "2026-03-25 14:30:00+00:00"
  },
  "api_key": "aid_proj_xR7kM2pQ9..."
}
Store the api_key securely.

The API key is shown once and cannot be retrieved again.

curl Example

curl -X POST https://api.agentsid.dev/api/v1/projects/ \
  -H "Content-Type: application/json" \
  -d '{"name": "my-app", "email": "dev@example.com"}'

Errors

CodeReason

429Rate limit exceeded

Security Model

Token Format (HMAC-SHA256)

AgentsID uses HMAC-SHA256 signed tokens rather than JWTs. Tokens are self-validating -- the server can verify authenticity without a database call for the signature and expiry checks.

aid_tok_<base64url(json_payload)>.<base64url(hmac_signature)>

Token Claims:
{
  "sub": "agt_7x9k2mNpQ4rS1tUv",  // Agent ID
  "prj": "proj_a1b2c3d4e5f6",    // Project ID
  "dby": "user_abc",            // Delegated by (human or parent agent)
  "iat": 1711324800,            // Issued at (Unix timestamp)
  "exp": 1711411200,            // Expires at (Unix timestamp)
  "jti": "tok_a1b2c3d4e5f6"     // Token ID (for revocation)
}

Validation Pipeline

Token validation follows a strict five-step pipeline. Every step must pass. Failure at any step short-circuits to denial.

1

Signature Verification No DB call

Recompute HMAC-SHA256 using the server's signing secret. Compare with constant-time comparison (hmac.compare_digest) to prevent timing attacks.

2

Expiry Check No DB call

Decode base64url payload and compare the exp claim against current Unix timestamp.

3

Project Ownership Check No DB call

Compare the prj claim against the authenticated API key's project. Prevents cross-project token use.

4

Revocation Check DB call

Look up the jti (token ID) in the database. If not found or revoked, reject. Fail-closed: unknown token IDs are treated as revoked.

5

Permission Check (optional) DB call

If a tool was specified, evaluate permission rules using the deny-first engine.

Permission Engine: Deny-First

The permission engine evaluates rules in three phases. Within each phase, rules are ordered by priority (highest first).

Tool CallParamsResultReason
delete_memoryanyDeniedMatches delete_* deny rule
save_memory{"category":"note"}AllowedMatches save_memory with valid condition
save_memory{"category":"secret"}DeniedCondition fails, default deny
save_memory(none)DeniedCondition requires params, fail-closed
search_memoriesanyAllowedMatches search_* allow rule
list_categoriesanyDeniedNo matching rule, default deny

Hash Chain Integrity

Each audit entry is cryptographically linked to its predecessor using SHA-256. The hash covers the project ID, agent ID, tool, action, result, delegated_by, params, error_message, and the previous entry's hash.

Tamper-evident.

Modifying, inserting, or deleting any audit entry breaks the chain from that point forward. The /audit/verify endpoint detects broken links.

The first entry in a project's chain uses the sentinel value "genesis" as its previous hash. Concurrent writes are serialized using SELECT ... FOR UPDATE to prevent chain forks.

Key Rotation

The AGENTSID_SIGNING_SECRET environment variable holds the server-side signing key. To rotate without invalidating active tokens:

  1. Set AGENTSID_SIGNING_SECRET_PREVIOUS to the old secret.
  2. Set AGENTSID_SIGNING_SECRET to the new secret.
  3. The server validates against the current secret first, then falls back to the previous secret.
  4. Once all tokens signed with the old secret have expired, remove AGENTSID_SIGNING_SECRET_PREVIOUS.

FAQ

What counts as a validation?▸
Every call to /api/v1/validate counts as one validation. The /check endpoint (which does not write to the audit trail) does not count. Token introspection via /introspect also counts as a validation.
Can I self-host AgentsID?▸
Yes. AgentsID is open source under the MIT license. The self-hosted option is a single FastAPI application backed by PostgreSQL. Enterprise customers get dedicated support for self-hosted deployments.
How are tokens different from JWTs?▸
AgentsID tokens use HMAC-SHA256 with a simpler structure than JWT. They include agent-specific claims (delegation chain, project scoping) and are designed specifically for agent-to-server authentication rather than general-purpose web auth. The signing secret never leaves the server.
What happens when my free tier limit is reached?▸
Validation requests will return HTTP 429 until the next billing cycle or until you upgrade. Your agents, tokens, and audit data are not affected -- they are just temporarily unable to perform new validations.
Is there a data processing agreement (DPA) available?▸
Yes. Enterprise customers can request a DPA. Contact sales@agentsid.dev for details.
How do I report a security vulnerability?▸
Email security@agentsid.dev with a description, reproduction steps, and potential impact. We acknowledge within 48 hours. Do not open public GitHub issues for security vulnerabilities.
