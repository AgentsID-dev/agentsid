## Archetype 1 — Methodology skepticism

> "How did you detect 'secretly'? Just a regex? What's the false-positive rate?"

Closed lexicon match against tool `description` fields: `secretly`, `silently`, `without notifying`, `without informing`, `hidden`, `covert`, `undetected`, and a handful of semantic equivalents. Detection rules are in `scanner/src/rules.mjs` — open source, MIT. All 460 flagged servers are individually enumerable at agentsid.dev/registry, so sampling N=50 at random and checking FPR is a ~15-minute exercise — I'd welcome it.

The conservative bias is toward precision: the lexicon doesn't catch metaphorical concealment, polite equivalents ("quietly update"), or non-English phrasings. Real concealment-intent prevalence is almost certainly higher than 460.

## Archetype 2 — "This is just a prompt injection alarm, no real exploit"

Fair framing pushback. Concealment-instructions are not "attacker sneaks payload into LLM context" — the developer shipped the instruction, and it fires every interaction. Whether the model follows it is model-dependent. We ran compliance tests (N=40/provider) for a related attack class (invisible-Unicode tag blocks): GPT-5.4 followed 100%, Claude flagged 100%, Gemini ignored 100%. Full methodology in the companion paper §4 (`invisible-ink.md`).

So: not an exploit in isolation. It's a trust violation — a tool description that, if obeyed, produces silent execution. The exploit paths are concealment + data-flow composition (e.g., servers that read external content AND execute code).

## Archetype 3 — Sample size / selection bias

Not a sample — it's a census. Every npm/PyPI package publicly resolvable as of 2026-04-17 whose manifest declares an MCP server: 15,983 total, 40,081 tools, 137,070 findings. No sampling frame, no survey weighting. Reproduction is one command: `npx @agentsid/scanner <package>`. Full methodology in `scanner/docs/census-2026/weaponized-by-design.md` §3.

## Archetype 4 — "What about Kerq / other directories?"

Different axis. Kerq scores reliability — uptime, latency, error rate. We score intent — deceptive language in descriptions, authorization-bypass patterns, Unicode smuggling, tool surface area. Kerq's own whitepaper: *"Kerq is not a security scanner. It doesn't evaluate code quality, vulnerability scanning, or compliance certifications."* Orthogonal. A tool can be 100 Kerq and F AgentsID.

## Archetype 5 — "Why letter grades instead of a score?"

Deliberate. A 0–100 number implies measurement precision we don't have — different categories (injection, permissions, auth, secrets, validation) aren't commensurable. A letter grade is a judgment: "this is the risk category." Rubric is public (`scanner/src/scoring.mjs`). Grades never change for payment — the integrity line is that you can't buy a better grade.

## Archetype 7 — Feature request: "Can I scan my private MCP server?"

Yes. `npx @agentsid/scanner <local-path>` scans a local directory by resolving its package manifest and spawning the server as a subprocess to enumerate tools. Works on private registries, local clones, and stdio servers. Nothing leaves your machine unless you explicitly submit the scan.

## Archetype 8 — "460 / 15,983 is ~3%. Not that bad?"

Fair point on ratio; the headline is about the concealment *type*, which is the most common Toxic Flow class but not the only one. Totals:

- 954 servers exhibit Toxic Flow behavior overall (6%)
- 1,332 servers grade F (8.3%)
- Of servers with 51+ tools, 94.8% grade F (n=115)
- Of servers with 1–5 tools, 1.8% grade F (n=14,259)

The tail, not the average, is the actual security surface. Aggregate percentages understate the concentration.

## Archetype 9 — "Show me RCE"

Concealment in isolation isn't RCE — it's trust violation. The RCE-class path is compositional: data-flow servers that (a) read external content and (b) execute code. Top example: `databricks-utils-mcp` (PyPI v1.26.0) — 41 tools, 42 critical findings, score 0. Per-tool breakdowns at agentsid.dev/registry. Paper §5.2 (concealment) + §5.5 (exfiltration chains) are the two types you'd pair to build a full chain.

## Archetype 10 — "Our server got an F — appeal process?"

Re-scan on every published release, so the fastest path to a grade change is shipping a fix and publishing — grades auto-update. No appeals-for-payment process, by design. If a detection rule is wrong on principle, open an issue: github.com/AgentsID-dev/agentsid. If a specific finding is wrong, include the package name and tool — we'll reproduce.

Claim flow (vendor claim of ownership, separate from grade dispute) at agentsid.dev/claim.

## Archetype 11 — "Why is this a protocol problem, not a model problem?"

Paper thesis: natural language is a structurally failed authorization layer. The LLM has no cryptographic way to distinguish operator policy from developer-authored tool descriptions from user input — they arrive as the same token stream. Fixing this at the model layer is a forever-battle (red-team/blue-team until heat death). Fixing it at the protocol layer — signed tool manifests, out-of-band permission scopes, description-hash pinning — is what MCP actually needs. We wrote a spec draft: `docs/tool-description-safety-spec.md`.

## Archetype 12 — "Did you disclose to the server authors?"

Depends on severity. For F-graded servers with critical findings, we're publishing aggregate data now and building a vendor-notify pipeline (part of the /claim flow). For any individual finding that constitutes a live, exploitable chain, we coordinate — happy to do that case-by-case; email on the paper. Open to feedback on the right process here.
